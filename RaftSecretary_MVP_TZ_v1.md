# RaftSecretary - MVP ТЗ v1

Дата: 2026-03-14

## 1. Цель

RaftSecretary - это локальное офлайн-приложение для секретаря соревнований по рафтингу.

Задача первой версии:
- надежно вести соревнование без интернета;
- быстро вводить результаты;
- автоматически считать места, очки и многоборье;
- строить сетку параллельного спринта;
- формировать печатные протоколы.

Приложение делается не как универсальная спортивная система, а как узкий рабочий инструмент для одного секретаря соревнований.

## 2. Основа MVP

Первая версия должна решать только 4 основные задачи:
- создать или открыть соревнование;
- завести команды;
- ввести результаты по дисциплинам;
- получить итоговые протоколы и таблицу многоборья.

Все решения в MVP принимаются в пользу:
- простоты;
- надежности;
- офлайн-работы;
- минимального числа действий для секретаря.

## 3. Пользователь

Основной пользователь один:
- секретарь соревнований.

Условия работы:
- один ноутбук;
- часто без интернета;
- результаты могут поступать по рации;
- работать нужно быстро, без сложной навигации и лишних экранов.

Синхронизация нескольких устройств в MVP не требуется.

## 4. Формат приложения

Приложение должно запускаться локально на компьютере пользователя и открываться в браузере.

Базовый принцип:
- пользователь распаковывает архив;
- запускает файл старта двойным кликом;
- приложение открывается в браузере;
- все данные хранятся локально.

## 5. Техническая рамка MVP

Для первой версии допускается следующая архитектура:
- backend: Python;
- локальная база: SQLite;
- frontend: HTML + простой JavaScript без тяжелой сборки;
- все зависимости должны работать офлайн;
- никакие CDN использовать нельзя.

Принцип хранения:
- один файл SQLite = одно соревнование.

Это нужно для:
- резервного копирования;
- переноса на другой компьютер;
- простой структуры данных.

## 6. Данные соревнования

Каждое соревнование содержит:
- название;
- дату;
- краткое описание;
- список активных дисциплин;
- список категорий, в которых проводится соревнование.

Категория - это отдельный зачет по сочетанию:
- класс судна;
- пол;
- возрастная группа.

Каждая категория считается отдельно: свои команды, свои результаты, свои места, свои очки, своя сетка параллельного спринта.

## 7. Категории

Поддерживаются:
- класс судна: `R4`, `R6`;
- пол: `мужские`, `женские`;
- возраст: `до 16`, `до 20`, `до 24`, `Кубок`, `Ветераны`.

Уточнение для проекта:
- смешанные экипажи учитываются как `мужские`.

В интерфейсе MVP отдельную категорию `смешанные` делать не нужно.

## 8. Данные команды

Для каждой команды хранятся:
- название команды;
- принадлежность: город, регион или страна;
- класс судна;
- пол категории;
- возрастная группа;
- стартовый номер;
- список участников с ФИО.

Для MVP стартовый номер принимается единым в рамках соревнования.

Отдельные стартовые номера по дисциплинам в первую версию не входят.

## 9. Дисциплины MVP

В MVP входят:
- спринт;
- параллельный спринт;
- слалом;
- длинная гонка;
- многоборье как автоматически считаемый итог.

Любая из первых четырех дисциплин может быть отключена при создании соревнования.

Приложение должно корректно работать при неполном наборе дисциплин.

## 10. Правила подсчета

Подсчеты в приложении должны опираться на официальный документ:
- правила вида спорта "Рафтинг", утвержденные приказом Минспорта РФ N 349 от 15 апреля 2022 года.

Если между удобством интерфейса и правилами возникает конфликт, приоритет у правил.

## 12. Статусы результата

Система должна поддерживать статусы:
- `Н/ФИН` - не финишировал;
- `Н/СТ` - не стартовал;
- `ДИСКВ/П` - дисквалификация на попытку или заезд;
- `ДИСКВ/С` - дисквалификация на соревнование.

Правила поведения:
- `Н/ФИН` - получает последнее место в дисциплине;
- `ДИСКВ/П` - получает последнее место в дисциплине;
- `Н/СТ` - не получает место и очки в дисциплине;
- `ДИСКВ/С` - место не присваивается, результаты аннулируются начиная с дисциплины, на которой произошла дисквалификация.

Если несколько команд имеют специальные статусы, сортировка должна учитывать правила и порядок старта.

## 13. Спринт

### 13.1 Суть

Раздельный старт, одна попытка.

### 13.2 Что вводит секретарь

Для каждой команды:
- чистое время;
- штраф за буи;
- штраф за неспортивное поведение;
- статус результата.

### 13.3 Что считает система

Система автоматически считает:
- итоговое время;
- место в дисциплине;
- очки в дисциплине;
- рейтинг для следующих этапов.

### 13.4 Правила

- при равенстве результата выше команда, стартовавшая раньше;
- статусные результаты обрабатываются по правилам соревнований.

## 14. Параллельный спринт

### 14.1 Общий принцип

Параллельный спринт проводится только после спринта.

Сетка строится на основе результатов спринта.

### 14.2 Этапы

Дисциплина состоит из:
- этапа отбора в основную сетку;
- основной сетки на выбывание.

Если число команд уже равно степени двойки, отборочный этап не нужен.

### 14.3 Требование к логике

Система должна:
- автоматически определить число команд во втором этапе;
- автоматически определить, кто попадает туда сразу;
- автоматически построить пары отборочного этапа;
- автоматически построить основную сетку;
- автоматически переносить победителей в следующий раунд.

### 14.4 Основа построения

Логика посева и сетки должна соответствовать официальным правилам и схемам Приложения N 4.

### 14.5 Что вводит секретарь

Для каждого заезда:
- результат двух команд;
- штрафы;
- статусы;
- при необходимости ручное подтверждение победителя.

### 14.6 Что считает система

Система должна:
- определять победителя заезда;
- распределять 1-4 места через финалы;
- распределять остальные места по времени последних заездов, как требуют правила.

## 15. Слалом

### 15.1 Общий принцип

Раздельный старт, две попытки.

В зачет идет лучшая попытка.

### 15.2 Порядок старта

Порядок старта строится по результатам спринта:
- от худшего к лучшему.

### 15.3 Настройка

Для каждой категории задается количество ворот.

В MVP допустимый диапазон:
- от 8 до 14 ворот по умолчанию;
- если нужно, администратор соревнования может задать другое количество вручную.

### 15.4 Что вводит секретарь

По каждой попытке:
- чистое время;
- штраф по каждому вороту.

Допустимые штрафы на воротах:
- `0`;
- `5`;
- `50`.

### 15.5 Что считает система

Система автоматически считает:
- сумму штрафов по попытке;
- итог попытки;
- лучшую попытку;
- итоговое место;
- очки за дисциплину.

### 15.6 Тайбрейки

При равенстве:
- сравнивается лучшая попытка;
- затем худшая попытка;
- затем меньший штраф в лучшей попытке;
- затем преимущество у ранее стартовавшего экипажа.

## 16. Длинная гонка

### 16.1 Общий принцип

Одна попытка.

Базовый режим в MVP:
- общий старт.

Допустимое расширение в рамках той же версии:
- групповой или раздельный старт, если это требуется условиями соревнования.

### 16.2 Порядок старта

Если применяется выбор стартовых позиций, он идет по текущему прямому рейтингу многоборья.

### 16.3 Что вводит секретарь

- чистое время;
- штрафы;
- статус результата.

### 16.4 Что считает система

- итоговое время;
- место;
- очки;
- обновленный общий зачет.

## 17. Многоборье

Многоборье считается автоматически как сумма очков за проведенные дисциплины.

Правила:
- больше очков = выше место;
- при равенстве очков выше команда с лучшим результатом в слаломе.

## 18. Экраны MVP

В первой версии нужны следующие экраны.

### 18.1 Главный экран

- список сохраненных соревнований;
- создать новое соревнование;
- открыть существующее соревнование;
- импортировать файл соревнования.

### 18.2 Настройка соревнования

- название;
- дата;
- описание;
- выбор категорий;
- выбор дисциплин;
- количество ворот для слалома.

### 18.3 Команды

- список команд;
- фильтры по категории;
- добавление;
- редактирование;
- удаление.

### 18.4 Жеребьевка и стартовый порядок

- автоматическое назначение стартовых номеров;
- ручная корректировка;
- выбор порядка старта для спринта.

### 18.5 Ввод результатов

Отдельные рабочие экраны:
- спринт;
- параллельный спринт;
- слалом;
- длинная гонка.

### 18.6 Рейтинг и многоборье

- очки по дисциплинам;
- сумма очков;
- итоговое место;
- фильтрация по категории.

### 18.7 Экспорт

- стартовый протокол;
- протокол дисциплины;
- итоговый протокол многоборья;
- резервный экспорт соревнования.

## 19. Экспорт

В MVP обязательно:
- печатная версия стартового протокола;
- печатная версия протокола результатов дисциплины;
- печатная версия итогового протокола многоборья;
- экспорт и импорт всего соревнования в JSON.

Требование MVP:
- сначала важна правильность структуры документа и читаемость;
- сложное визуальное оформление не требуется.

Если прямой PDF-экспорт окажется нестабильным на старых системах, допустим промежуточный вариант:
- печатная HTML-страница с сохранением в PDF через браузер.

## 20. Дизайн интерфейса

Требования к интерфейсу:
- очень компактный;
- высокий контраст;
- хорошо читается на солнце;
- цифры времени и очков заметны сразу;
- минимум декоративных элементов;
- без лишних модальных окон;
- основной сценарий должен укладываться в таблицы и простые формы.

Для MVP достаточно:
- светлой темы;
- русского языка интерфейса.

Темная тема и английский язык не входят в первую версию.

## 21. Надежность данных

Это критическая часть MVP.

Обязательные требования:
- данные пишутся сразу в локальную базу;
- соревнование можно открыть повторно без потери данных;
- можно сделать резервный экспорт в JSON;
- после каждого изменения результаты должны быть пересчитаны сразу.

Желательно включить в первую версию:
- журнал изменений;
- защита от случайного удаления;
- простую отмену последнего действия для результатов.

Если это не помещается в первую реализацию, сначала обязательно делать:
- подтверждение опасных действий;
- автоматическую резервную копию JSON.

## 22. Что не входит в MVP

Не входит в первую версию:
- облачная работа;
- синхронизация нескольких устройств;
- авторизация пользователей;
- мобильная версия;
- адаптация под планшеты;
- импорт из Excel;
- интеграция с внешними системами хронометража;
- отдельная категория смешанных экипажей;
- английский язык интерфейса;
- темная тема;
- сложная ролевая модель;
- установка через системный инсталлятор.

## 23. Ключевые продуктовые решения MVP

Для первой версии фиксируются такие решения:
- один пользователь;
- один локальный файл SQLite на соревнование;
- русский язык;
- светлая тема;
- стартовый номер единый на все соревнование;
- смешанные экипажи учитываются как мужские;
- интерфейс ориентирован на ноутбук;
- приоритет у надежности и правил, а не у красоты.

## 24. Критерии готовности первой версии

Версию можно считать рабочей, если секретарь может:
- создать новое соревнование;
- занести категории и команды;
- провести спринт;
- автоматически получить сетку параллельного спринта;
- провести слалом;
- провести длинную гонку;
- увидеть корректный общий зачет;
- экспортировать печатные протоколы;
- закрыть приложение, открыть снова и увидеть все сохраненные данные.

## 25. Открытые вопросы на следующую стадию

Эти вопросы не блокируют MVP, но должны быть отдельно подтверждены перед реализацией:
- нужен ли в первой версии групповой старт для длинной гонки как полноценный режим;
- нужен ли CSV-импорт команд уже в MVP или можно отложить;
- нужен ли полноценный журнал действий в первой версии;
- какой именно формат печатных протоколов обязателен в первую очередь.
